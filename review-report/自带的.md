### 变更概述

本次提交主要做了两点改动：

1. 在 `ArticleNoContent` 列表模型中新增了 `share_password` 字段，使文章列表接口也返回分享密码。
2. 调整 `ShareArticleResource.post` 中分享密码校验逻辑，引入 `is_hashed` 参数，区分前端传入的是原始密码还是已经哈希后的值。

## 各维度评审意见

### 1. 逻辑正确性

**问题 1：分享密码校验逻辑整体正确，无明显逻辑 bug**  
- **级别：建议**  
- **位置**  
  - [article_manager.py#L246-L260](/Users/maplewan/keep-going/codes/1_LiFocus/lifocus-backend-v2/app/controllers/article/article_manager.py)

**分析**

新逻辑：

```python
if article.share_password:
    parser = reqparse.RequestParser()
    parser.add_argument('password', type=str, required=False)
    parser.add_argument('is_hashed', type=bool, required=True)
    args = parser.parse_args()

    if not args.get('password'):
        return {'code': 400, 'message': ARTICLE_ERROR_MESSAGE['SHARE_PASSWORD_REQUIRED']}, 400
    if args['is_hashed']:
        if args['password'] != article.share_password:
            return {'code': 401, 'message': ARTICLE_ERROR_MESSAGE['SHARE_PASSWORD_INCORRECT']}, 401
    else:
        if not verify_password_with_salt(article.share_password, args['password']):
            return {'code': 401, 'message': ARTICLE_ERROR_MESSAGE['SHARE_PASSWORD_INCORRECT']}, 401
```

- 若 `is_hashed == True`：直接比较客户端传入的哈希值与数据库存储的哈希值。
- 若 `is_hashed == False`：调用 `verify_password_with_salt(article.share_password, args['password'])` 进行密码校验。

从纯逻辑角度，这两种分支都能实现「密码正确才通过」的目标，没有明显的条件遗漏或分支错误，是对原先错误逻辑（使用未导入的 `hash_password_with_salt`）的修正和优化。

**建议**

- 将 `is_hashed` 的语义在接口文档中明确说明（如：`True` 表示前端直接传递后端之前返回的哈希串），避免误用。

### 2. 安全性

**问题 2：文章列表接口新增返回分享密码字段，扩大了敏感信息暴露面**  
- **级别：警告**  
- **位置**  
  - [article_api_model.py#L23-L35](/Users/maplewan/keep-going/codes/1_LiFocus/lifocus-backend-v2/app/controllers/article/article_api_model.py)

```python
article_no_content_model = article_ns.model('ArticleNoContent', {
    'id': fields.String(description='文章 ID'),
    'category_id': fields.Integer(description='文章分类 ID'),
    'type': fields.String(description='文章类型'),
    'title': fields.String(description='文章标题'),
    'status': fields.String(description='文章状态'),
    'is_shared': fields.Boolean(description='文章是否共享'),
    'share_password': fields.String(description='文章分享密码'),
    'is_deleted': fields.Boolean(description='文章是否删除'),
    'create_time': fields.String(description='文章创建时间'),
    'update_time': fields.String(description='文章更新时间'),
    'category': fields.Nested(category_model, description='文章分类')
})
```

**问题描述**

- 原先 `ArticleNoContent` 模型中不包含 `share_password`，列表接口不会返回分享密码。
- 本次变更后，文章列表接口（如 `CategoryArticleResource` GET/POST）也会随列表返回 `share_password` 字段（即数据库中存储的“盐 + 哈希”值）。
- 即使该值是哈希+盐处理后的密码，它依然是「登录/访问凭据」，属于敏感认证信息。一般情况下不应该在 API 响应中回传给客户端，尤其是列表接口中批量返回，会显著扩大泄露面。

**为什么是新问题**

- 详情接口（`article_with_content_model`）以前就返回 `share_password`，这是既有设计问题；  
- 但这次新增的是列表模型中的 `share_password`，使原本不暴露的列表接口现在开始暴露分享密码，这是**本次提交新增的风险面**，满足“新引入问题”的条件。

**修复建议**

至少有两种较合理的方案：

1. **不在列表接口返回具体分享密码，只返回是否设置密码的布尔标记**（推荐）  

   ```python
   article_no_content_model = article_ns.model('ArticleNoContent', {
       'id': fields.String(description='文章 ID'),
       'category_id': fields.Integer(description='文章分类 ID'),
       'type': fields.String(description='文章类型'),
       'title': fields.String(description='文章标题'),
       'status': fields.String(description='文章状态'),
       'is_shared': fields.Boolean(description='文章是否共享'),
       # 删除 'share_password' 字段
       'has_share_password': fields.Boolean(description='是否设置分享密码'),
       'is_deleted': fields.Boolean(description='文章是否删除'),
       'create_time': fields.String(description='文章创建时间'),
       'update_time': fields.String(description='文章更新时间'),
       'category': fields.Nested(category_model, description='文章分类')
   })
   ```

   同时在模型转换/序列化时，把 `has_share_password` 映射为 `bool(article.share_password)`。

2. 如果确实有必要在某些管理界面展示分享密码，应限制为**只在专门的管理详情接口**返回，且该接口必须有严格鉴权，不建议在通用列表接口中返回。

### 5. API 兼容性

**问题 5：请求参数由 `have_hashed` 改为 `is_hashed`，存在明确的接口兼容性破坏风险**  
- **级别：严重**  
- **位置**  
  - [article_manager.py#L248-L260](/Users/maplewan/keep-going/codes/1_LiFocus/lifocus-backend-v2/app/controllers/article/article_manager.py)

**问题描述**

- 原逻辑中：

  ```python
  parser.add_argument('have_hashed', type=bool, required=True)
  ```

  客户端是通过 `have_hashed` 参数告知后端「密码是否已经哈希」。

- 本次提交中改为：

  ```python
  parser.add_argument('is_hashed', type=bool, required=True)
  ```

- 如果线上已有客户端（Web/桌面/移动端等）仍然发送 `have_hashed` 字段：
  - 新代码中不会再解析 `have_hashed`；
  - `is_hashed` 是 `required=True`，缺失时 `parser.parse_args()` 会直接触发参数错误，返回 400；
  - 这是一个**确定性的**接口行为变动，会导致旧客户端在未同步更新的情况下无法访问分享文章接口。

**为什么这是“新问题”且非推测性**

- 旧代码显式支持 `have_hashed`，新代码完全移除此参数并新增一个必填参数 `is_hashed`。
- 即使无法得知具体有多少客户端在使用，但从接口协议角度看，这显然是一个破坏性修改，与「保持向后兼容」的常规 API 设计原则相悖，因此可以判定为高置信度问题。

**修复建议**

在过渡阶段建议兼容两种参数名，再逐步下线旧参数。例如：

```python
parser = reqparse.RequestParser()
parser.add_argument('password', type=str, required=False)
# 兼容旧参数名
parser.add_argument('have_hashed', type=bool, required=False, dest='is_hashed')
parser.add_argument('is_hashed', type=bool, required=False)
args = parser.parse_args()

# 统一得到 is_hashed 值
is_hashed = args.get('is_hashed')
if is_hashed is None:
    # 如果两个字段都没传，返回参数错误
    return {'code': 400, 'message': ARTICLE_ERROR_MESSAGE['PARAM_ERROR']}, 400
```

或保持解析时两者都可接受，内部统一使用 `is_hashed` 变量名来处理。待前端完全切换后，再移除对 `have_hashed` 的支持。

## 变更总结（Summary of Changes）

- 在 `ArticleNoContent` API 模型中新增 `share_password` 字段，使文章列表接口也返回分享密码信息。
- 在 `ShareArticleResource.post` 中，新增 `is_hashed` 参数，替代原有的 `have_hashed` 参数，用于区分前端传入的密码是原文还是哈希。
- 分享文章的密码校验逻辑由“错误的重复哈希比较 + verify”调整为根据 `is_hashed` 选择直接比较哈希或调用 `verify_password_with_salt`，逻辑更合理且避免使用未导入函数。
- 整体上，是一次围绕文章分享密码相关功能的接口与校验逻辑调整，同时引入了列表接口返回分享密码及参数名变更的兼容性问题。

## 总结与建议

1. **优先修复**：  
   - 对分享文章接口的参数名变更（`have_hashed` → `is_hashed`）进行向后兼容处理，至少在一个版本周期内同时支持两种参数名，避免线上旧客户端请求全部失败。（严重）

2. **安全性改进**：  
   - 重新评估在列表接口中返回 `share_password` 的必要性，建议改为返回 `has_share_password` 这类非敏感标记，而不是具体的（盐+哈希）值，以减少敏感凭据在系统中的暴露面。（警告）
